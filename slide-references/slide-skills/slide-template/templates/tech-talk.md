# Tech Talk 技術簡報模板

## 模板概覽

| 項目 | 說明 |
|------|------|
| **適用場景** | 技術分享會、架構說明、新技術介紹、Code Review |
| **聽眾** | 技術人員、工程師、開發者 |
| **核心目標** | 傳遞技術知識、分享實戰經驗、展示解決方案 |
| **推薦主題** | Tech Innovation, Midnight Galaxy |
| **時長範圍** | 10-45 分鐘 |

---

## 結構模板

### 10 分鐘精簡版 (8-10 頁)

```
1. 封面 (30s)
   - 標題：[問題] → [解決方案]
   - 講者、日期、場合

2. 問題/挑戰 (1.5 min)
   - 我們遇到什麼問題？
   - 為什麼這很痛？
   - 業界常見做法

3. 我們的方案 (3 min)
   - 核心概念
   - 技術選型理由
   - 架構圖

4. 關鍵實作 (2.5 min)
   - 程式碼片段 / 設定
   - 踩過的坑

5. 成果展示 (1.5 min)
   - Demo / Before-After
   - 效能數據

6. 結論與 Q&A (1 min)
   - 學到的教訓
   - 未來方向
```

### 20 分鐘標準版 (15-18 頁)

```
1. 封面 (30s)

2. Agenda 大綱 (30s)
   - 簡述 4-5 個段落

3. 背景脈絡 (2 min)
   - 專案背景
   - 技術環境
   - 團隊組成

4. 問題定義 (2 min)
   - 遇到的挑戰
   - 影響範圍
   - 為什麼現有方案不夠

5. 方案探索 (3 min)
   - 評估過的選項
   - Trade-off 分析
   - 最終選擇理由

6. 架構設計 (4 min)
   - 整體架構圖
   - 關鍵模組說明
   - 資料流 / 控制流

7. 實作細節 (4 min)
   - 核心程式碼
   - 設定與配置
   - 整合方式

8. 踩坑紀錄 (2 min)
   - 遇到的問題
   - 如何解決
   - 避坑指南

9. 成果與效益 (1.5 min)
   - 效能對比
   - 維護性改善
   - 團隊回饋

10. 結論與未來 (1 min)
    - Key Takeaways
    - 下一步計畫
    - 資源連結

11. Q&A
```

### 45 分鐘深度版 (35-40 頁)

擴展標準版，增加：
- 詳細技術背景 (5 min)
- 多個實作模組分開講解 (10 min)
- Live Demo 環節 (5 min)
- 效能調優細節 (5 min)
- 延伸討論與互動 (5 min)

### 程式碼頁

```
┌─────────────────────────────────────────────────────────────┐
│  Cache Strategy Implementation                              │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ public async Task<T> GetOrSetAsync<T>(             │   │
│  │     string key,                                      │   │
│  │     Func<Task<T>> factory,                          │   │
│  │     TimeSpan? expiry = null)                        │   │
│  │ {                                                    │   │
│  │     var cached = await _redis.GetAsync<T>(key);     │   │
│  │     if (cached != null) return cached;              │   │
│  │                                                      │   │
│  │     // 防止快取雪崩：加入隨機過期時間                  │   │
│  │     var jitter = Random.Shared.Next(60, 300);       │   │
│  │     var finalExpiry = (expiry ?? _defaultExpiry)    │   │
│  │         .Add(TimeSpan.FromSeconds(jitter));         │   │
│  │                                                      │   │
│  │     var value = await factory();                    │   │
│  │     await _redis.SetAsync(key, value, finalExpiry); │   │
│  │     return value;                                    │   │
│  │ }                                                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  💡 重點：加入 jitter 避免同時過期                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 敘事技巧

### 開場方式

**方式 A：從痛點開始**
> 「你有沒有遇過這種情況：凌晨三點被 alert 叫醒，發現 DB 快被打爆了...」

**方式 B：從數據開始**
> 「上個月，我們的 API 平均回應時間是 850ms，P99 更是超過 2 秒...」

**方式 C：從問題開始**
> 「當你的系統需要支援 10 倍流量時，第一個瓶頸會在哪裡？」

### 轉場語

- 「有了這個架構後，讓我們看看實際怎麼寫...」
- 「理論講完了，來看看踩過的坑...」
- 「程式碼是這樣，但實際上線後呢？」

### 結尾方式

**方式 A：Key Takeaways**
> 「今天記住三件事：1. 快取不是萬靈丹 2. Jitter 很重要 3. 監控先行」

**方式 B：Call to Action**
> 「這個方案的程式碼在這個 repo，歡迎 PR。有問題 Slack 找我。」

## 常見問題預測

| 問題 | 建議回答 |
|------|----------|
| 為什麼不用 XXX？ | 評估過，主要考量是... 不過 XXX 在 YYY 情境下更適合 |
| 效能數據怎麼測的？ | 用 k6/JMeter，10 分鐘穩定測試，取 P50/P99 |
| 有遇到什麼坑嗎？ | 最大的坑是 XXX，解法是... |
| 團隊怎麼學的？ | 先 pair programming，再 code review，最後寫 doc |
